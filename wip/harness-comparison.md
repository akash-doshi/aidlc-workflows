# AI-DLC Harness Review: Claude Code vs Kiro Implementation

> Reviewed from the perspective of an end-user developer using the framework as-is today.

---

## Architecture & Composability

| Metric | Claude Code Implementation | Kiro Implementation | Verdict |
|--------|---------------------------|---------------------|---------|
| **Composability (adding/removing stages)** | Stage definitions are `.md` files inside `.claude/aidlc-common/stages/<phase>/`. Each stage declares frontmatter (`scopes:`, `sensors:`, `for_each:`, etc.). The compiled `stage-graph.json` (produced by `aidlc-graph.ts compile`) must be regenerated. Scopes are also files (`.claude/scopes/aidlc-<name>.md`) and membership is declared in stage frontmatter. Adding a stage = write the .md + tag scopes + run `compile` + run `runner-gen write`. | Stage definitions are plain `definition.md` files in `src/stages/<stage-name>/`. No compilation step needed at runtime — the orchestrator LLM reads `stage-graph.md` (a markdown dependency table) and uses it to compose workflows conversationally. Adding a stage = write a `definition.md` + add a row to `stage-graph.md` + run the build script (just file copy). | **Kiro is simpler for an end-user.** Adding a stage requires no compilation/validation toolchain, no frontmatter schema adherence, and no multi-file edits. Claude Code's approach is more rigorous but demands toolchain knowledge. |
| **Ease of changing execution order** | The engine reads the compiled `stage-graph.json` and the scope determines which stages run and in what order. To reorder: edit the stage's `order` number in frontmatter + recompile. Users can also use `--stage` or `--phase` flags to jump to a specific point. The graph is strict and deterministic. | The orchestrator composes the workflow conversationally with the human. The human can say "do wireframes before stories" and the LLM reorders. The dependency graph explicitly states: "The human may specify their preferred order... rearrange stages accordingly." The `workflow.json` is built incrementally, stage-by-stage, with human approval each time. | **Kiro is far more flexible.** Order is negotiated at runtime with the human rather than pre-compiled. Claude Code requires CLI flags or frontmatter edits. |
| **Reordering example: "wireframes before requirements"** | Requires either CLI flags (`--stage` to jump), manual stage skipping at gates, or editing stage order numbers + recompiling the graph. The engine owns sequencing — it walks stages in compiled order per scope. The user can override within the engine's model (jump/skip) but can't conversationally resequence. | User says it in the composition conversation. The orchestrator reads the flexible dependency graph, confirms both stages can run from intent alone, and builds the workflow in the requested order. Zero friction. | **Kiro wins decisively.** |
| **Many front doors (different people, different starting points)** | Everyone in the team shares one compiled graph with one fixed order per scope. A designer wanting mockups first hits stages 1.1–1.5 before reaching 1.6. Personal workflow preferences require per-person scope overrides or manual jumps. | Each intent gets its own composed workflow. The stage-graph supports multiple entry points per stage ("Can consume from: intent, wireframes, stories..."). Person A starts with wireframes, Person B starts with requirements — both compose naturally. | **Kiro is purpose-built for this.** The flexible dependency graph + conversational composition supports "many front doors, same methodology." |

## Performance & Cost

| Metric | Claude Code Implementation | Kiro Implementation | Verdict |
|--------|---------------------------|---------------------|---------|
| **Wall-clock latency (overhead)** | Heavy toolchain overhead per step: each "next" call shells out to `bun .claude/tools/aidlc-orchestrate.ts next`, then `report`. Hooks fire on Write/Edit (6 event types × ~20ms bun startup each). The engine is deterministic and fast (pure TypeScript, no LLM for routing). The real bottleneck is the LLM itself — the massive SKILL.md (~500+ lines) + conductor persona + stage file are loaded into context every turn. | Minimal tooling — only 2 tools (`workspace-setup.js`, `process-checker.js`). No engine binary calls between stages. One hook (postToolUse on sub-agent). Orchestration decisions are made by the LLM reading skills. Less per-turn overhead from tool orchestration, but the orchestrator must reason about "what's next" in prose. | **Tie with trade-offs.** Claude Code has higher mechanical overhead but the engine prevents LLM from wasting tokens on routing. Kiro has near-zero tool overhead but the LLM does all routing reasoning. |
| **Throughput (stages per minute)** | Pre-approved permissions (settings.json grants Read/Write/Bash/Task/etc). No per-call permission prompts. The swarm system enables parallel unit-of-work execution via `Task` calls or Dynamic Workflows. Construction stages can fan out to N workers simultaneously. | Each stage is sequential: orchestrator → sub-agent (persona) → return. No parallelism mechanism. Each sub-agent invocation goes one at a time. No swarm/fan-out capability. | **Claude Code is faster for multi-unit construction.** The swarm with parallel `Task` calls can build N units concurrently. Kiro is strictly serial. |
| **Token consumption (same requirement)** | Very high base context: CLAUDE.md (~200 lines) + SKILL.md (~500 lines) + conductor persona + stage protocol + knowledge files + rules (5-layer chain) are loaded into context. The 1M-context window (Opus 4.8) is intentionally used. 10 hooks inject output. On the plus side, the deterministic engine means the LLM doesn't waste tokens reasoning about routing. | Moderate context: orchestration skill ~50 lines, stage-execution skill ~100 lines, workflow-composition ~100 lines. Stage definitions ~30 lines. Personas ~20 lines YAML. Skills are loaded on-demand per agent via `resources` (only the relevant ones). However, the LLM must reason about routing, composition, and ordering (which Claude Code offloads to the engine). | **Kiro consumes fewer tokens.** The per-turn context is dramatically smaller. Claude Code front-loads everything for determinism. Kiro's "read only what you need now" approach is token-efficient. |
| **LLM spend (cost model)** | Configured for Opus 4.8 (1M context) — the most expensive model available. Parallel swarm workers each consume their own token budget. A full `enterprise` scope runs 32 stages, each with full context loaded. Very expensive. | No model pinned — uses Kiro's default. Smaller per-turn context = less per-API-call cost. Sequential execution = one active LLM call at a time. Fewer stages in default path (typically 5-8 selected). Significantly cheaper per intent. | **Kiro is substantially cheaper.** Claude Code optimizes for thoroughness at high cost. Kiro is lean-by-default. |

## Workspace & Multi-Project

| Metric | Claude Code Implementation | Kiro Implementation | Verdict |
|--------|---------------------------|---------------------|---------|
| **Day-2: Second intent in same workspace** | Running `/aidlc <new description>` starts a new workflow. State is workspace-scoped (`aidlc-docs/aidlc-state.md`). Only ONE active workflow exists at a time. The `--init` guard refuses to scaffold over existing state. A new intent requires `--init --force` to wipe, or the old workflow must be `done`. | Each intent gets its own numbered directory: `org-ai-kb/aidlc-docs/intent-001-<slug>/`. The workspace-setup script auto-increments (`getNextIntentNumber()`). Multiple intents coexist natively — state, audit, and artifacts are fully isolated per intent. | **Kiro is significantly better.** It natively supports multiple intents in the same workspace without conflict. Claude Code's single-workflow model requires explicit lifecycle management. |
| **Multi-repo support** | Framework installs inside one project (`.claude/` at project root). `$CLAUDE_PROJECT_DIR` is the single anchor. Worktree tooling helps isolate parallel units within that repo, but orchestrating across independent repos requires separate sessions or nesting everything under one parent. MCP servers provide external reach but don't solve cross-repo state coordination. | Workspace-first architecture: `org-ai-kb/` holds per-repo knowledge and documentation, intents live at the workspace level (not repo level), and you bring code in from Git into the workspace. An intent can naturally span multiple repos because state and artifacts are scoped to the intent, not to a repo. Reverse-engineering runs per-repo and stores results in `org-ai-kb/`. | **Kiro is better.** The workspace-over-repo architecture is a genuine multi-repo design. Claude Code is fundamentally single-project. |

## Flexibility & Customization

| Metric | Claude Code Implementation | Kiro Implementation | Verdict |
|--------|---------------------------|---------------------|---------|
| **"We write requirements like this"** | Three layers of persistent rules + practices-discovery stage. The practices-discovery stage interviews the team and writes practices into `aidlc-team.md` and `aidlc-project.md`. The §13 learnings ritual accumulates corrections. Rules are loaded into context for every stage via the 5-layer chain. But there's no mechanism to swap the stage template itself — the stage's `## Steps` are framework-owned. | Templates are plain files you edit. Each stage has a `templates/` directory. The work-method skill says: "Use the templates in the stage's `templates/` directory as the starting format." Want Gherkin requirements? Replace the template. Additionally, workflow-composition is conversational — just tell the LLM your preferred format. | **Kiro is more directly flexible.** Swap a template = change the output. Claude Code routes through an indirect rule system. |
| **Persistence of custom formats** | Excellent. Once practices-discovery affirms a format or the learnings loop captures a correction, it persists in `aidlc-team.md` forever and applies to every future workflow. The rule chain ensures it's always in context. | Poor persistence today. Templates are part of the framework installation (`.kiro/stages/`). The build script does a clean `rmrf` then copies — so customizations are overwritten on rebuild. No learnings system means custom formats don't accumulate automatically. | **Claude Code is better for long-term format persistence.** Its rules accumulate; Kiro's templates are fragile on rebuild. |
| **Granularity of customization** | Very granular: per-stage (stage rules, future), per-phase (phase rules), per-project, per-team, per-org. Each layer inherits and overrides. Sensors validate that outputs conform. | Coarse: per-template, per-stage-definition. No layered rule system. No validation that output matches custom format. | **Claude Code is more granular** for teams with nuanced format preferences. |
| **Natural feel of customization** | Ceremony-heavy: run practices-discovery stage, go through affirmation gate, or manually edit rule files, or wait for learnings ritual. Requires understanding the rule system. | Natural: say "we write requirements like this" in conversation and the LLM adapts. Zero ceremony for one-off. But no persistence mechanism for recurring preferences. | **Kiro feels more natural in the moment.** Claude Code's investment pays off over time but has higher ceremony. |

## Robustness & Observability

| Metric | Claude Code Implementation | Kiro Implementation | Verdict |
|--------|---------------------------|---------------------|---------|
| **State management & recovery** | Deterministic, tool-owned state: `aidlc-state.ts` manages transitions, `aidlc-bolt.ts` handles fork/merge. Markdown-based state file with typed fields. PreCompact hook validates consistency. Session hooks log boundaries. Recovery protocol exists. Audit is append-only with locking (mkdir-based). | JSON-based state (`state.json`, `audit.json`, `workflow.json`). State transitions are LLM-executed — the work-method skill tells the persona "set status to X in state.json." No locking. No validation hooks. Only the `process-checker.js` hook checks file existence. | **Claude Code is more robust.** Tool-owned state with validation prevents drift/corruption. Kiro delegates state writes to the LLM, which can fail or hallucinate. |
| **Auditability & observability** | 10 hooks: audit logger, session start/end, statusline, subagent tracking, state validation, sensor firing, runtime compilation. StatusLine shows progress in terminal. Comprehensive event taxonomy (GATE_APPROVED, STAGE_STARTED, SWARM_UNIT_CONVERGED, etc.). Self-learning captures decisions as rules. | Audit is a simple JSON append managed by the orchestrator LLM. One hook. No statusline. No sensors. No learnings persistence. The process checker validates outputs exist — not content quality or state consistency. | **Claude Code is far superior.** Rich telemetry, comprehensive events, defensive self-verification. Kiro has minimal observability. |
| **Error handling & resilience** | Multiple layers: PreCompact state validation, Stop hook for session end, validated directives (malformed = hard error), sensor-based verification, swarm anti-tamper checks, recovery protocol. The engine never fails silently. | The process-checker runs post sub-agent and checks "do declared output files exist?" That's the only automated check. No formal recovery mechanism documented. | **Claude Code is more resilient.** Defense-in-depth vs reliance on LLM correctness with minimal guardrails. |
| **Self-improvement / learning** | §13 learnings ritual: surfaces interpretations/deviations/tradeoffs after each gated stage, persists confirmed learnings to rules files. Can install custom sensors. Rules persist across workflows and accumulate team knowledge over time. | No self-learning mechanism. No rules persistence between sessions. Each new intent starts from the same baseline. Human corrections don't carry forward automatically. | **Claude Code is clearly better.** The framework gets better over time for a team. Kiro resets every time. |

## Human Experience

| Metric | Claude Code Implementation | Kiro Implementation | Verdict |
|--------|---------------------------|---------------------|---------|
| **Human control / approval gates** | Every non-bootstrap stage has a gate. Tri-mode interaction (guided, self-guided, chat). Questions in markdown with A-E + X options. Can grant Construction autonomy (swarm vs gated). `--test-run` for CI auto-approval. | Every stage has a 16-step state machine with explicit "presented" gate. Per-stage autonomy (`full`/`supervised`) in workflow.json. Human can override anytime. Workflow composition is inherently interactive. | **Tie.** Claude Code offers finer autonomy graduation and CI mode. Kiro offers more natural conversational interaction. |
| **User onboarding / setup complexity** | Requires: bun installed and on PATH for non-interactive shells, AWS Bedrock credentials configured, region set, model IDs pinned. 10 hooks, 25 tools, compiled graph. Copy `.claude/` into project. Run `--doctor` to validate. Multiple prereqs that can each fail. | Requires: Node.js (standard). Run `npm run build` (file copy). Drop `.kiro/` into workspace. Two tools, one hook. No external services required. | **Kiro is much easier to set up.** Minutes vs potentially hours of configuration. |
| **IDE integration** | Terminal-based (Claude Code CLI). StatusLine for progress. No native IDE panel integration. Works anywhere Claude Code runs (terminal, CI, SSH). | Native Kiro IDE integration: hooks fire in-IDE, agents render in IDE panels, skills load via `skill://` protocol. Designed for the IDE experience. | **Kiro is better for IDE users.** Claude Code is better for terminal/CI users. |

## Document Efficiency (Human Digestibility)

| Metric | Claude Code Implementation | Kiro Implementation | Verdict |
|--------|---------------------------|---------------------|---------|
| **Number of output files per design stage** | Application Design produces 5 files: `components.md`, `component-methods.md`, `services.md`, `component-dependency.md`, `decisions.md`. Functional Design produces 4: `business-logic-model.md`, `business-rules.md`, `domain-entities.md`, `frontend-components.md`. 9 files across two design stages a human must read. | Domain Design produces 2 files: `components.yaml` + `components.md` (human-readable view). Functional Design produces 4-7 but explicitly separates machine-readable (YAML) from human-readable: "This is the human-readable view. entities.yaml and rules.yaml are the source of truth." | **Kiro is better.** Explicit separation of "source of truth for AI" vs "summary for humans." Claude Code produces all-markdown with no "start reading here" signal. |
| **Document density / verbosity** | Stage instructions demand "Detailed algorithms, workflows, data transformations, processing sequences, and decision trees." Depth controls thoroughness but even Standard produces dense prose. No built-in summarization layer. | Templates are intentionally terse — tables, one-line workflow steps, mermaid diagrams. Template header: "Minimum structure. Sections may be omitted with rationale or extended as needed." | **Kiro produces leaner documents.** Templates bias toward tabular summaries and diagrams over prose. |
| **"Where do I start reading?"** | Human gets "Review path: `aidlc-docs/inception/application-design/`" — a directory of 5 files. They must figure out which is overview vs detail. | Explicit: `functional-spec.md` = human reads this. `entities.yaml` + `rules.yaml` = AI reads this. `components.yaml` = blueprint that carries forward. | **Kiro is better.** Clear separation. |
| **Total artifact sprawl** | A full `feature` workflow (32 stages × multiple files each) can produce 50+ files across the directory tree. No consolidated "here's everything you need to know" document. | Fewer stages (typically 5-8 used), each producing 2-4 files. The blueprint-identity model means `components.yaml` carries forward and accumulates — one canonical document rather than scattered truth. | **Kiro produces less total documentation.** Living blueprint vs distributed files. |
| **Diagram-first vs prose-first** | Stages mention "Include Mermaid diagrams" but primary output is text-heavy markdown. Diagrams are additions to prose. | Templates lead with mermaid diagrams (ER diagrams, component diagrams). Tables are the primary format. Prose is minimal. | **Kiro is more visual by default.** Teams that won't read documents might at least look at a diagram. |
| **Customizability of output verbosity** | Controlled by Depth (Minimal/Standard/Comprehensive) which affects stage body thoroughness, not output template. No "give me a one-pager" option. | Replace `functional-spec.md` template with a one-page summary and the persona follows it. The owner's `plan.md` decides which outputs are relevant — not all are mandatory. | **Kiro is easier to tune for brevity.** Swap a template = shorter output. |

## Summary

**Pick Kiro if** you want something lightweight, token-efficient, easy to set up, naturally interactive (conversational workflow composition), cheap to run, flexible on execution order, supportive of different working styles ("many front doors"), multi-repo friendly, and you produce documents that humans will actually read. Best for teams working in the Kiro IDE on moderately complex projects where sequential execution is acceptable.

**Pick Claude Code if** you need maximum rigour, parallelism (swarm), deterministic reproducibility, enterprise audit trails, self-learning that accumulates over time, multi-scope support, sensor-based verification, or you're running in CI/automated environments where the LLM mustn't freestyle its routing.

**The core architectural difference:** Claude Code externalizes all routing decisions into a deterministic TypeScript engine (the LLM is a skilled executor following typed directives), while Kiro keeps the LLM as the reasoning orchestrator (it reads markdown and decides what to do). This is the fundamental trade-off — control vs simplicity, rigour vs flexibility, cost vs capability.
